# Claude plugins & skills

A small collection of Claude Code skills, distributed as a single plugin
marketplace. Each one ships in **two forms** assembled from one source of truth:

- a **standalone skill** — works in any Claude Code project, installed as a plugin;
- a **BMad persona agent** — the same capability wrapped as a [BMad](https://github.com/bmad-code-org/BMAD-METHOD) module (a named persona with the menu/customization plumbing), installed via the BMad installer.

| Domain | Standalone skill | BMad agent |
| --- | --- | --- |
| **arc42 documentation** | `arc42-documentation` | Gernot — *arc42 Documentation Architect* |
| **Music domain** | `music-domain-expert` | Adam — *Music Domain Expert* |

## Install — standalone skills (any Claude Code project)

This repository is a plugin marketplace. Add it once, then install whichever
plugins you want:

```
/plugin marketplace add mschuerig/claude-plugins
/plugin install arc42-documentation@mschuerig
/plugin install music-domain-expert@mschuerig
```

(The marketplace id — the part after `@` — is the `name` field in
`.claude-plugin/marketplace.json`.)

When you publish a new skill later, add it under `plugins/`, list it in
`.claude-plugin/marketplace.json`, and existing users pick it up with
`/plugin marketplace update mschuerig`.

## Install — BMad agents

The BMad agents under `bmad/` are installed by the BMad installer (not the
Claude Code plugin system), **directly from GitHub** — no manual clone or build.

**Install each agent via its subdirectory URL** (recommended). BMad then reads
that subdirectory's own marketplace and shows exactly **one** agent — no clutter:

```
# arc42 Documentation Architect (Gernot)
https://github.com/mschuerig/claude-plugins/tree/v1.0.0/bmad/arc42-documentation-architect
# Music Domain Expert (Adam)
https://github.com/mschuerig/claude-plugins/tree/v1.0.0/bmad/music-domain-expert
```

Drop the `/tree/v1.0.0` ref to track the default branch instead of a pinned
release.

> **Do not point the installer at the bare repo root for agents.** The root
> marketplace must also list the two standalone Claude Code skills (that file is
> the *only* place Claude Code's `/plugin` can read them). BMad lists every
> plugin it finds, so a repo-root URL shows **all four** entries — the two agents
> (suffixed *“(BMad Agent)”*) and the two skills, which are not BMad modules and
> will fail if selected. The subdirectory URLs above avoid this entirely.

**Why the generated files are committed.** The BMad installer (6.8.x) clones the
repo but does **not** run `npm install`, so a plain clone must already be
complete. The BMad modules' `references/` and `scripts/` are therefore committed
to git even though they are *generated* from the canonical sources by
`tools/build.mjs`. CI rebuilds them and fails on any drift, so they can never go
stale — the canonical sources remain the single source of truth (see below).

> Local-path installs also work (`source: custom`, `localPath:
> /path/to/claude-plugins/bmad/<module>`).

### How resolution works (and why it once failed)

The 6.8.x installer scans the **repo-root** `.claude-plugin/marketplace.json` and
matches a single-skill plugin whose `skills[0]/assets/module.yaml` `code` equals
the BMad module code (its "Strategy 3"). The root marketplace therefore carries
a dedicated entry per agent (`arc42-documentation-architect`,
`music-domain-expert-agent`) pointing at the agent skill dir. These live
alongside the two standalone-skill entries; Claude Code's `/plugin` listing shows
all four, with distinct names so nothing collides. `tools/check-bmad-marketplace.mjs`
(run in CI) guards this wiring; `tools/acceptance-bmad-resolver.mjs` drives the
real BMad resolver end-to-end against a simulated clone.

## Repository layout & the single-source model

```
.claude-plugin/marketplace.json   # marketplace index — lists the standalone plugins
plugins/                          # standalone skills (CANONICAL, committed, install-ready)
  arc42-documentation/
    skills/arc42-documentation/
      SKILL.md
      references/                 # ← single source of truth for arc42 knowledge
  music-domain-expert/
    skills/music-domain-expert/SKILL.md
bmad/                             # BMad persona-agent forms
  _shared/scripts/                # ← single source of truth for BMad install scripts
  arc42-documentation-architect/
    agent-.../
      SKILL.md                    # persona shell (Gernot)
      customize.toml
      assets/
      references/  scripts/       # GENERATED by build, COMMITTED (installer needs them)
  music-domain-expert/
    agent-.../
      SKILL.md  customize.toml  assets/
      scripts/                    # GENERATED by build, COMMITTED
Makefile                          # `make build` assembles the BMad forms
```

**The standalone skill is the source of truth.** The BMad agent reuses the same
material:

- `make build` copies the arc42 `references/` (capability + knowledge files)
  from the standalone skill into the BMad module verbatim — so the arc42 domain
  knowledge lives in exactly one place.
- `make build` distributes the identical BMad install scripts from
  `bmad/_shared/scripts/` into each module.

To change arc42 behavior or knowledge, edit the files under
`plugins/arc42-documentation/skills/arc42-documentation/` and re-run
`make build`. Never edit the generated `references/` / `scripts/` under `bmad/`.

### Known limitation: music is not single-sourced

The two arc42 forms share files cleanly. The two **music** forms do not: the
standalone skill expresses its domain knowledge as prose in `SKILL.md`, while
the BMad agent expresses it as persona fields in `customize.toml`. These are
different representations of the same expertise, so they're maintained in
parallel by hand. Mechanically deduplicating them would require a content
redesign and is intentionally out of scope.

## Building

The build is one dependency-free Node script, `tools/build.mjs`:

```
make build        # or: npm run build  — assemble the BMad forms from canonical sources
make clean         # remove the generated BMad artifacts
```

The generated `references/`/`scripts/` are **committed** (the BMad installer
needs a complete clone — it does not run the build). After editing any canonical
source, re-run `make build` and commit the result. An npm `prepare` hook runs the
same script on `npm install` as a convenience for local dev.

CI (`.github/workflows/ci.yml`) rebuilds on every push/PR, **fails on any drift**
between committed artifacts and canonical sources, runs the BMad marketplace
guard (`tools/check-bmad-marketplace.mjs`), validates the SKILL.md
frontmatter and JSON manifests, and cuts a GitHub Release on `v*` tags.

### Cutting a release

Tag a commit and push the tag; CI creates the release. BMad's pinned/stable
channels resolve against pure-semver tags (`vMAJOR.MINOR.PATCH`):

```
git tag v1.0.0 && git push origin v1.0.0
```
